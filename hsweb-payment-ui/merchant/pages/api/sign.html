交互流程

 

1、商户首先要对需要加密的参数进行加密，然后对请求报文做签名，最后请求支付公司网的网关。

2、商户收到网关响应和通知后，首先对参数进行验签，确保参数没有被篡改。验证通过后，对加密的参数进行解密。

 

一、签名与验签

请求报文，响应报文都采用相同的规则：对发送或接收到的所有参数，除去sign参数外，都是需要参与计算待签名字符串的参数。

 

生成待签名字符串

根据不同的报文场景，确定好待签名的数据项，采用如下格式进行组装（以下举例说明，具体参数参考接口说明）：

Json parameters = {
    
   "amount":1,
     "productId":"f03b2c8b5ad3fb0d7981acc5a0d6dffb",
     "merchantId":"1073055130872942593",
     "orderId":"67979e59b31637bf47ca87e51ea915bd",
    "requestId":"9511c3e6456a87aa302963aa41ddf8fc",
    "ip":"183.67.60.146",
    "channel":"weixin-h5",
     "notifyUrl":"xxx.com",
    "returnUrl":"",
    "productName":"产品名称"

    
};

对数组里的每一个成员按 字符ASCII码 的顺序排序，若遇到相同首字母，则看第二个字符，以此类推。排序完成之后，再把所有数组值以“&”字符链接起来，这串字符串就时待签名的字符串，例如：

amount=1&channel=weixin-h5&ip=183.67.60.146&merchantId=1073055130872942593¬ifyUrl=xxx.com&orderId=67979e59b31637bf47ca87e51ea915bd&productId=f03b2c8b5ad3fb0d7981acc5a0d6dffb&productName=%E4%BA%A7%E5%93%81%E5%90%8D%E7%A7%B0&requestId=9511c3e6456a87aa302963aa41ddf8fc&returnUrl=

注意：

1、没有值的参数无需传递，也无需包含到待签名数据中；

2、签名时将字符转化成字节流时指定字符集全部采用UTF-8;

 

验签

 

摘要方式签名主要使用MD5，采用标准的摘要算法实现。算法原则如下：

  1. 签名明文: 签名和验签的明文为代签字符串后接商户安全码（我们这里命名为secretKey商户开户时已邮件发送给商户联系人），请注意待签字符串与secretKey间用& 符号链接。

    例如:

    • 代签字符串为：key1=val1&key2=val2,
    • secretKey为: c9cef22553afujh64b04a012f9cb8ea9,
    • 则计算签名的明文为：key1=val1&key2=val2&sign=c9cef22553afujh64b04a012f9cb8ea9

  2. 签名计算：首先转换签名明文(P)为UTF-8的字节数组（byte[]）,通过标准摘要算法计算出摘要值的字节数组形式（byte[]），然后转换摘要的字节数组为Hex（小写字母的16进制形式）的字符串格式(S)则为摘要签名。 伪代码公式为: S=Hex(MD5(P.getBytes(“UTF-8”)))

    摘要签名计算出的值作为请求报文的sign参数值一并请求网关。

  3. 验证签名：验证签名是在商户端收到网站的同步响应报文或通知报文后，对这些报文进行签名ß验证。原则为，使用“2. 签名计算”的方法对报文计算的签名值与响应或通知报文中的sign参数值（服务器端计算的签名）进行字符串比较，如果相同则表示验证签名通过，报文在传输过程中未被篡改。

 

二、请求服务

  1. 构造请求数据：商户根据网关提供的接口规则，通过程序生成得到签名结果及要传输给网关的数据集合。

  2. 发送请求数据：把构造完成的数据集合，通过页面链接跳转或表单提交的方式传递给网关。

    请求实例：

    POST /open-api/payment/query HTTP/1.1
    Content-Length: 141
    Content-Type: application/x-www-form-urlencoded; charset=UTF-8
    Host: localhost:8080
    Connection: Keep-Alive
    User-Agent: Apache-HttpClient/4.5.5 (Java/1.8.0_181)
    Accept-Encoding: gzip,deflate
    ​
    merchantId=1064831713916805121&paymentId=1068418317722030081&requestId=0a18fe020b3543db0a81800b215113fd&sign=931575f49991161ddbe209b31f67e274
    ​

  3. 服务处理：网关得到这些集合后，会先进行安全校验等验证，一系列验证通过后便会处理这次发送过来的数据请求。

  4. 同步响应数据：对于处理完成的请求，网关以同步响应的方式返回处理结果和数据，响应数据在同一个http连接的response报文的body体中返回，以JSON格式进行传输。

  5. 响应数据处理：商户系统同步接收到http响应后，获取Http响应的body体中的JSON报文体，根据具体服务接口定义的结构解析获取数据后，进行商户系统的业务处理。
